# Находки: feasibility-спайк локального MLX-ASR (Replika)

- **Дата:** 2026-07-15
- **Статус:** спайк завершён
- **Тип:** итоговые находки + go/no-go вердикт
- **Родительские документы:** [`design.md`](2026-07-15-mlx-asr-feasibility-spike-design.md), `architecture.md`
- **Стенд:** Apple Silicon, macOS 26.5.2, 36 GB unified memory, Xcode Swift 6.3.3 (swift-tools-version 6.0)

Документ фиксирует результаты запуска бенч-матрицы (Task 10) на реальных
клипах и даёт итоговый вердикт по стратегии «обёртка над `speech-swift`».
**Ноль транскрипт-текста, ноль личных имён** — все качественные заметки
намеренно обобщены.

## 2. Качество (на слух)

Проверено на слух на обоих клипах (`call_60s.wav`, q4 и q8) — **без**
копирования текста куда-либо:

- **RU читается корректно** на обоих quant-тирах: связная русская речь,
  без выпадений слов и без явной деградации на слух между q4 и q8.
- **Метки спикеров совпадают с реальной структурой разговора**: ровно 2
  различных спикера (`S0`/`S1`), чередование меток по ходу звонка
  соответствует поочерёдной речи двух сторон разговора (двусторонний
  телефонный звонок).
- **Word-timestamps** (через `ForcedAligner` → `SegmentBuilder`)
  монотонны, укладываются в границы аудио, сегменты по паузам режутся
  правдоподобно относительно услышанного темпа речи — точечная сверка на
  слух подтверждает совпадение таймкодов сегментов с реальными паузами.
- На длинном клипе (9.4 мин, q4) качество **оценить не удалось** — конвейер
  не выдал ни одного сегмента (см. §5), сравнивать не с чем.

## 3. Build / runtime reality

- **Собирается как чистый SwiftPM CLI** на Swift 6 (`swift-tools-version:
  6.0`, компилятор Swift 6.3.3) — без Xcode app-бандла, без entitlements.
  `swift build` (полный пакет, все таргеты) стабильно чистый, без
  warnings, ~7–8 с на инкрементальной пересборке.
- **Sortformer-диаризация работает из голого CLI** (`swift run
  replika-spike`) без app-контекста — подтверждено обоими бенч-запусками
  на 60-секундном клипе (2 спикера определены и вмёржены в финальные
  сегменты оба раза, на q4 и на q8).
- **MLX `mlx.metallib`-ловушка (реальная CI/дистрибуционная проблема)**:
  MLX не собирает шейдеры сам по себе — `mlx.metallib` должен существовать
  в `.build/<triple>/debug/` (генерируется отдельным скриптом
  `build_mlx_metallib.sh` из `speech-swift`, требует Xcode Metal
  Toolchain), и **отдельно** копироваться внутрь `.xctest`-бандла для
  `swift test` (скрипт `speech-swift` знает только имена бандлов
  собственного пакета, не нашего `ReplikaPackageTests`) — иначе плейн
  `swift run`/`swift test` падает в рантайме с
  `MLX error: Failed to load the default metallib`, при этом сама *сборка*
  (`swift build`) линкуется и завершается успешно — ошибка невидима до
  первого реального запуска. На этом стенде метallib уже присутствовал в
  обоих нужных местах с предыдущих задач (Task 8), поэтому Task 10 не
  повторно наступал на эту ловушку — но это гарантированно всплывёт на
  свежем чек-ауте/CI-раннере.
- **Размеры загрузок моделей** (измерено `du -sh` на кэше
  `~/Library/Caches/qwen3-speech/models/aufklarer/` на этом стенде):

  | модель | размер на диске |
  |---|---|
  | `Qwen3-ASR-0.6B-MLX-4bit` | 680 MB |
  | `Qwen3-ASR-1.7B-MLX-4bit` | 2.1 GB |
  | `Qwen3-ASR-1.7B-MLX-8bit` | 2.3 GB |
  | `Qwen3-ForcedAligner-0.6B-4bit` | 938 MB |
  | `Qwen3-ForcedAligner-0.6B-8bit` | 1.2 GB |
  | `Sortformer-Diarization-CoreML` | 239 MB |

  Полный q4-стек для продукта (ASR 1.7B-4bit + aligner 0.6B-4bit +
  Sortformer) — **~3.3 GB**; полный q8-стек — **~3.7 GB**. Это больше, чем
  «~2.1 GB на q4-стек» из брифа/плана — та оценка, похоже, учитывала
  только ASR-модель, без aligner и диаризатора.
- Кэш моделей лежит в `~/Library/Caches/qwen3-speech/`, не в наш
  `Application Support` — известный, уже задокументированный зазор (design
  §7), переносится в следующий срез (см. §6 ниже).

## 4. speech-swift API-дельты vs план

Все дельты уже были реконсилированы в Task 2/8 и подтверждаются здесь без
изменений (метрики/сигнатуры, без транскрипт-контента):

- **Диаризация — `SortformerDiarizer`, не `DiarizationPipeline`.**
  `DiarizationPipeline` реально существует, но это typealias на
  `PyannoteDiarizationPipeline` (embedding+clustering движок), не
  Sortformer. Провайдер использует `SortformerDiarizer.fromPretrained()` /
  `.diarize(audio:sampleRate:config:) -> DiarizationResult`.
- **Quant выбирается подстрокой в `modelId`, не отдельным аргументом.**
  Нет `quantBits:`/`variant:` параметра нигде в API — `fromPretrained(modelId:)`
  сам детектит размер/битность по подстроке repo-id (`"1.7B"` → large,
  `"8bit"`/`"4bit"` → битность). Провайдер реализует это через приватный
  `Quant → modelId` маппинг.
- **У forced aligner нет 1.7B-варианта** — `ForcedAlignerVariant` имеет
  только `mlx4bit`/`mlx5bit`/`mlx8bit`/`bf16`, все на базе 0.6B. Aligner
  остаётся 0.6B для обоих quant-тиров ASR — неизбежное, задокументированное
  несоответствие размеров ASR/aligner, не недосмотр.
- **Базовые сигнатуры ASR/aligner совпали с планом** — async/throws-профиль
  и имена полей (`.text`, `.startTime`, `.endTime`) не потребовали
  изменений; только `Float`→`Double`-конверсии на границе с
  `ReplikaCore.Word`/`SpeakerTag` (обе величины `Float` у `speech-swift`,
  `Double` у наших доменных типов).

## 5. Находка: единичный вызов `transcribe()` не масштабируется на длинное аудио (новое, Task 10)

Это новая находка, вскрытая именно длинным (9.4 мин) прогоном — не была
видна на 60-секундном клипе ни в одной из задач 1–9.

**Наблюдение.** На полном 9.4-минутном клипе (q4) CLI завершился без
ошибки (`exit 0`), с правдоподобным wall-clock (~41–43 с) и peak RAM
(~6.9 GB), но **вывел ноль спикер-размеченных сегментов** — только строку
`Warning: no words found in text` от forced aligner (это сообщение
печатается им, когда переданный текст не содержит ни одного слова). Оба
повтора прогона дали идентичный результат — воспроизводимо, не разовый
сбой.

**Причина (подтверждена чтением исходников, не догадка).**
`SpeechSwiftProvider.transcribe` передаёт **весь** массив сэмплов клипа
одним вызовом в `Qwen3ASRModel.transcribe(audio:sampleRate:language:context:)`
— конвейер (Task 1–9, как задизайнено и как в самом `speech-swift`) не
делает чанкинг/оконное разбиение длинного аудио. У `Qwen3ASRModel`
жёсткий `maxTokens = 448` на весь вызов, плюс отдельная
«длинновходовая» эскалация decoding-опций при `audioDurationSeconds > 15 с`
(bump `noRepeatNgramSize` до 3, антидеградационная мера для длинных
клипов) — но она рассчитана на клипы порядка «больше 15 секунд», а не на
9+ минут. На 562.8 с аудио модель на этом стенде вернула текст, который
forced aligner интерпретировал как не содержащий ни одного слова — далее
`SegmentBuilder.build(words: [])` детерминированно даёт пустой список
сегментов, и `SpeakerMerger` мёржить не с чем — поэтому CLI печатает ноль
строк, не падая и не предупреждая на уровне пользователя (только служебный
`os.Logger`/stderr-принт из самого `speech-swift`).

Отдельно подтверждено: `AudioLoader` корректно прочитал и ресемплировал
**весь** 562.8-секундный `.m4a`-клип (расчёт `audioSeconds` из
`RTF × wall` даёт ≈563 с, что совпадает с реальной длительностью клипа
по `afinfo`) — обрыв происходит не на загрузке аудио, а строго в
ASR/aligner-звене конвейера.

**Что это значит.** Диаризация (Sortformer) технически может отработать
на полном клипе нормально — но её результат никогда не всплывает на CLI,
потому что финальные сегменты строятся из ASR-слов, а не из сырых
диаризационных тегов напрямую; при пустом ASR-выводе диаризация
эффективно «немая» для пользователя. Это отдельный, более мелкий
зазор наблюдаемости (нет пути показать диаризацию независимо от ASR),
не блокирующий сам по себе.

**Вывод для продукта.** Текущий, спайковый способ вызова (один
`transcribe()` на весь файл) **не годится** для реальных звонков
продолжительностью в несколько минут — типичной длины для целевого
сценария продукта. Это архитектурное ограничение самого способа вызова
(и текущей формы decoding-эскалации в `speech-swift`), не баг конкретной
задачи 1–9 и не quant-специфичный эффект. Нужен слой чанкинга (VAD-based
сегментация + скользящее окно/overlap-and-merge) поверх единичного
`transcribe()`-вызова — это поднимается до **самого приоритетного** зазора
в вердикте ниже, выше по срочности, чем изначально запланированные
"live/streaming mode" и "model-manager" срезы.

## 6. Находка по пиковой памяти

Полный конвейер (1.7B-4bit ASR + 0.6B aligner + Sortformer + внутренние
MLX-буферы), все модели резидентны одновременно в течение жизни процесса:

- **q4: peak RAM ≈ 7.3–7.5 GB** (7488 MB в этом прогоне; 7363–7528 MB —
  диапазон по трём независимым прогонам, включая Task 9).
- **q8: peak RAM ≈ 8.6 GB** (8619 MB) — примерно на **1.1 GB больше**, чем
  q4, ожидаемо (8-битные веса тяжелее 4-битных на ASR и aligner
  одновременно).
- На длинном клипе (q4, 9.4 мин, но с обрывом конвейера — см. §5) peak
  RAM был **ниже** (~6.9 GB) — правдоподобно объясняется тем, что
  генерация текста остановилась рано (пустой/короткий выход), поэтому
  decoder-буферы не выросли так же, как на полном 60-секундном
  транскрипте; не интерпретировать это как «длинное аудио дешевле по
  памяти» в общем случае.

**Практический вывод по классам железа** (не изменился относительно
Task 9, переподтверждён здесь): комфортно на **24–32 GB** unified memory;
**впритык** на **16 GB** (весь пайплайн + ОС + остальное приложение может
не влезть одновременно); **не жизнеспособно на 8 GB**.

RTF, измеренный CLI, **включает разовую загрузку моделей с диска**
(`fromPretrained` × 3 модели) — это не чистый inference-RTF. Task 2
(bare ASR-probe, тёплый кэш, без aligner/diarizer) видел warm-RTF **~0.30**
(~3.3× быстрее реального времени) на 0.6B-модели — заметно лучше, чем
RTF 1.83–2.20 из бенч-таблицы §1. Разница ожидаема и не противоречива:
разные знаменатели (load+diarize+ASR+align против чистого ASR-инференса
на прогретой модели меньшего размера).

## 7. Вердикт

**Стратегия «обёртка над `speech-swift`» — подтверждена для коротких
клипов, с одной блокирующей оговоркой для длинных.**

- **Да**, `speech-swift` реально заводится на этой машине нативно
  (SwiftPM CLI, Swift 6, без Python/подпроцессов), корректно
  транскрибирует RU-речь, отдаёт монотонные word-timestamps и правильно
  диаризует/мёржит метки спикеров — все критерии приёмки design-документа
  (§3 design.md, пп. 1–4, 6) выполнены на клипе до 60 с включительно, на
  обоих quant-тирах.
- **Но**: единичный вызов `transcribe()` без чанкинга **не работает** на
  9.4-минутном реальном звонке (0 сегментов на выходе, см. §5) — критерий
  приёмки 5 (замеры на «мультиспикер-клипе») формально выполнен на уровне
  метрик (RTF/RAM измерены), но содержательно длинный клип **не
  протранскрибирован**. Для продукта, где типичный звонок — это минуты,
  а не десятки секунд, это блокирующий зазор, не полировка.

**Рекомендуемый quant по умолчанию: q4.** На 60-секундном клипе q4 и q8
неотличимы на слух по качеству RU и по корректности спикерных меток; q4
даёт заметно меньший peak RAM (7.5 GB против 8.6 GB, на ~1.1 GB меньше) и
меньший дисковый след (2.1+938M против 2.3G+1.2G ≈ на ~0.5 GB меньше).
q8 в этом конкретном прогоне показал более короткий wall-clock (27.3 с
против 32.7 с), но при полностью прогретом кэше на обоих — это, вероятнее
всего, шум планировщика/дискового кэша между прогонами, а не устойчивое
"q8 быстрее"; RAM — более стабильный дифференциатор, и он в пользу q4.
Совпадает с уже зафиксированным дефолтом в acceptance criterion 1.

**Зазоры, переносимые в следующий срез (по приоритету):**

1. **[Новое, самый приоритетный] Чанкинг длинного аудио.** Единичный
   `transcribe()`-вызов не масштабируется за пределы примерно
   короткого клипа (работает уверенно на 60 с; ломается на 9.4 мин) —
   нужен VAD-based слой сегментации/overlap-merge поверх текущего
   провайдера прежде, чем продукт сможет обрабатывать реальные звонки.
   Это меняет приоритет относительно исходного плана — этот пункт теперь
   важнее, чем live/streaming mode ниже, потому что он блокирует даже
   файловый (не live) сценарий на реалистичной длине записи.
2. **Перенос кэша моделей** из `~/Library/Caches/qwen3-speech/` в наш
   `Application Support` (через `cacheDir:`/`QWEN3_CACHE_DIR`) —
   зафиксировано в design.md §7, актуально без изменений.
3. **Model-manager / реестр / манифесты / докачка** — за нас сейчас
   работает голый `fromPretrained()`, продукту нужен полноценный
   менеджер (прогресс, отмена, повтор при обрыве сети — уже наблюдался
   один транзиентный network timeout в Task 8).
4. **Live/streaming режим** — вне scope спайка по дизайну; `speech-swift`
   имеет стриминговые примитивы, не задействованные здесь.
5. **Aligner language plumbing** — `align()` вызывается с
   `language: "English"` по умолчанию (см. Task 2); эмпирически
   безвредно для RU (whitespace-delimited fallback-путь), но ломает
   CJK word-align — нужно прокидывать реальный язык явно.
6. **AudioLoader error-surface hardening** — уже отмечено в progress.md
   (Task 4): unwrapped `file.read`, Int64→UInt32 truncation trap на
   огромных файлах, нет error-path тестов.
7. **MLX `mlx.metallib`-packaging для дистрибуции** — см. §3; сейчас это
   ручной build-скрипт + ручное копирование в test-бандл, нужно
   автоматизировать до CI/дистрибуции конечного продукта.
